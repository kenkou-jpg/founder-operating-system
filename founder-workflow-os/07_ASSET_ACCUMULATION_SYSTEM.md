# 07 — Asset Accumulation System

> 「新しいアプリを作るたびに何が資産として積み上がるか」を体系化する。
> 一人 Founder の最大の強みは、複利で蓄積する資産。

---

## 資産蓄積の思想

一人 Founder は時間が限られている。
しかし **一度作った資産は、次のアプリを速く・良く・安く作れるようにする**。

新しいアプリを作るたびに:
- コードは再利用できる（テンプレート）
- 判断基準は再利用できる（Founder OS）
- ブランドは強くなる（信頼の蓄積）
- ナレッジは深まる（ドメイン知識）
- 顧客基盤は広がる（メールリスト等）

これが Founder にとっての「複利」。

---

## 資産カテゴリー

### 1. アプリ資産（Product Assets）

**定義:** 実際に動いているプロダクトそのもの

**蓄積方法:**
- 各アプリのコードベース・設計・ドメイン知識
- プロダクトが生む MRR・ARR
- ユーザーデータ（適切な同意のもと）

**次アプリへの活用:**
- `templates/APP_STARTER_TEMPLATE/` として再利用
- 共通コンポーネントの抽出
- 実績・信頼として新アプリの PR に活用

**管理場所:** `portfolio-os/[PRODUCT].md`

---

### 2. ブランド資産（Brand Assets）

**定義:** Founder・プロダクトへの認知・信頼・評判

**蓄積方法:**
- 継続的な note 記事・X 投稿
- プロダクトの品質と誠実さ
- ユーザーコミュニティとの関係

**次アプリへの活用:**
- 新アプリの告知が「既存ファン」に届く
- メディア露出・インタビューの機会
- App Store での信頼（デベロッパー実績）

**管理場所:** `brand-os/` + note プロフィール + X プロフィール

---

### 3. SEO 資産（SEO Assets）

**定義:** 検索エンジンからの自然流入を生む資産

**蓄積方法:**
- 有益なコンテンツの継続発信（note・独自ドメインブログ）
- ドメインオーソリティの蓄積
- キーワードランキングの維持

**次アプリへの活用:**
- 関連キーワードでの新アプリ紹介記事
- 既存コンテンツからの内部リンク
- 読者への新アプリ告知

**管理場所:** `brand-os/` のコンテンツ戦略

---

### 4. note 資産（Content Assets）

**定義:** note に蓄積したコンテンツと読者との関係

**蓄積方法:**
- 開発記録・プロダクト思想の記事
- 技術解説・ビジネス考察
- 読者（フォロワー）との継続的な関係

**次アプリへの活用:**
- 新アプリ launch 記事が既存読者に届く
- プロダクト思想を共感する読者がいる
- SEO 資産との相乗効果

**管理場所:** note アカウント + `brand-os/` のコンテンツガイド

---

### 5. X（Twitter）資産（Social Assets）

**定義:** X でのフォロワー・エンゲージメント・影響力

**蓄積方法:**
- 開発過程の発信（build in public）
- ドメイン知識の発信
- コミュニティとの継続的な交流

**次アプリへの活用:**
- Launch ツイートが既存フォロワーに届く
- ユーザーインタビュー候補の調達
- 口コミ・シェアの起点

**管理場所:** X プロフィール + `brand-os/` のソーシャル方針

---

### 6. メールリスト資産（Email List Assets）

**定義:** メールアドレスを提供してくれたユーザー・読者との関係

**蓄積方法:**
- 各アプリのニュースレター登録者
- note・X からのメール登録誘導
- 価値あるコンテンツ・情報の定期提供

**次アプリへの活用:**
- 新アプリのβテスター募集
- Launch 時の初期ユーザー獲得
- ユーザーインタビュー依頼

**管理場所:** メールサービス（Mailchimp / ConvertKit 等）

---

### 7. GitHub 資産（GitHub Assets）

**定義:** GitHub 上に蓄積したコード・設計・実績

**蓄積方法:**
- コードの公開（OSS化できるもの）
- 設計ドキュメントの公開
- `founder-operating-system` リポジトリ自体

**次アプリへの活用:**
- 再利用可能なライブラリ
- スターが「信頼の証明」になる
- エンジニアコミュニティからの認知

**管理場所:** GitHub プロフィール + 各リポジトリ

---

### 8. Research 資産（Research Assets）

**定義:** 調査・分析・ユーザーインタビューから得たドメイン知識

**蓄積方法:**
- `research-os/insights/` へのインサイト追加
- ユーザー行動パターンの分析記録
- 市場調査の蓄積

**次アプリへの活用:**
- 同一ドメインの次アプリで即座に活用
- インタビュー候補（過去ユーザー）の再活用
- Imaging Agriculture のような研究事業への展開

**管理場所:** `research-os/insights/` + `knowledge-os/`

---

### 9. Template 資産（Template Assets）

**定義:** 再利用可能なテンプレート・ワークフロー・プロセス

**蓄積方法:**
- `templates/APP_STARTER_TEMPLATE/` の継続改善
- `development-os/pr-generator-os/` の 11_PR_RESPONSIBILITY_REGISTRY.md 蓄積
- `founder-workflow-os/` 自体の改善

**次アプリへの活用:**
- 新アプリ立ち上げ時間の短縮（72時間以内目標：v1.0）
- 品質の再現性
- テンプレート事業としての販売

**管理場所:** `templates/` + `template-business-os/`

---

### 10. AI Prompt 資産（AI Prompt Assets）

**定義:** AIを活用した作業の効率化に使えるプロンプト集

**蓄積方法:**
- 繰り返し使うプロンプトの記録
- `development-os/pr-generator-os/` の PR プロンプトテンプレート
- `templates/COUNCIL_TEMPLATE/` のペルソナ会議プロンプト

**次アプリへの活用:**
- 設計・PRプロンプトの即時再利用
- AI エージェントへのシステムプロンプトとして
- Founder OS v2.0（AI 統合）の素材

**管理場所:** `development-os/pr-generator-os/` + `templates/COUNCIL_TEMPLATE/` + 専用プロンプト管理ファイル（今後追加）

---

### 11. Operating System 資産（OS Assets）

**定義:** Founder Operating System 自体

**蓄積方法:**
- 各アプリの知見を OS にフィードバック
- 判断基準・テンプレート・チェックリストの改善
- `governance/CHANGELOG.md` での進化の記録

**次アプリへの活用:**
- これがあることで、次のアプリは初日から同じ品質基準でスタートできる
- OS が成熟するほど、新アプリの立ち上げコストが下がる

**管理場所:** `founder-operating-system/` リポジトリ全体

## 資産蓄積チェック（アプリリリース後 3ヶ月ごと）

```
□ アプリ資産: コードのどの部分がテンプレート化できるか
□ ブランド資産: コンテンツ発信は継続しているか
□ SEO資産: インデックスされているコンテンツが増えているか
□ note資産: 今月何本記事を書いたか
□ X資産: フォロワー・エンゲージメントの変化
□ メールリスト: 今月何人増えたか
□ GitHub資産: スター数・コントリビューション
□ Research資産: インサイトを何件追加したか
□ Template資産: テンプレートを何件更新したか
□ AI Prompt資産: 新しいプロンプトを記録したか
□ OS資産: Founder OSに何を反映したか
```

---

## 資産蓄積の優先順位

すべての資産を同時に最大化はできない。
フェーズごとの優先順位:

| フェーズ | 最優先資産 | 次点 |
|---------|----------|------|
| Idea〜MVP | アプリ資産・OSasset | Template資産 |
| PMF確認中 | Research資産・ブランド資産 | メールリスト |
| Growth | ブランド資産・SEO・X | メールリスト |
| 安定運営 | Template資産・OS資産 | すべてバランス良く |
| Long-term | OS資産・Research資産 | note・SEO |

---

## 資産の「複利」が効く時期

```
1プロダクト目（1〜2年）:
  資産がほぼゼロから始まる。
  しかしFounder OSが育ち始める。

2プロダクト目:
  APP_STARTER_TEMPLATEで立ち上げが速くなる。
  ブランド・メールリストに既存読者がいる。
  PR Registryのパターンがある。

3プロダクト目以降:
  OSが成熟し、スタートコストが大幅減。
  ブランドが「シリアル Founder」として認知される。
  各資産が相互に補完し合う。
```
