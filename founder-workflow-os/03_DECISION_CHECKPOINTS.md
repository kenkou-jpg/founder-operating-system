# 03 — Decision Checkpoints

> Founder が必ず判断するべき意思決定ポイント。
> 流れに乗って「なんとなく進む」を防ぐ。
> 各ポイントで意識的に決断し、`governance/DECISION_LOG.md` に記録する。

---

## 意思決定の原則

1. **決断を先延ばしにしない** — 決断が必要なポイントに来たら、その場で決める
2. **決断の根拠を記録する** — `governance/DECISION_LOG.md` に「なぜそう決めたか」を書く
3. **データで決める** — 感情・焦り・外圧で決めない（`OPERATING_PRINCIPLES.md` #12）
4. **覆せる判断と覆せない判断を区別する** — 覆せない判断ほど慎重に

## CP-01 — 市場に進むか（Problem Discovery 後）

**タイミング:** S02 Problem Discovery 完了後

**問い:**
- この課題は実在するか？（一次情報で確認済みか）
- 市場は十分な規模があるか？
- 自分がこの課題を解くのに適しているか？

**選択肢:**
- **Go** → S03 Business Strategy へ進む
- **Pivot** → 課題仮説を修正して S02 やり直し
- **Stop** → このアイデアを棄却する

**記録先:** `governance/DECISION_LOG.md`

---

## CP-02 — ビジネスモデルの決定（Business Strategy 完了後）

**タイミング:** S03 Business Strategy 完了後

**問い:**
- サブスク / フリーミアム / B2B2C どれにするか？
- 初期価格はいくらか？
- 無料プランを設けるか？

**判断基準:**
- `business-os/` の価格設計原則
- 競合の価格帯と自分の差別化

**選択肢:**
- 選定したモデルで進む
- 2案を並走させ、PMF確認後に決定（リスクを取りたくない場合）

**記録先:** `governance/DECISION_LOG.md` + `governance/BINDING_DECISIONS.md`

---

## CP-03 — 設計凍結（Architecture 完了後）

**タイミング:** S11 Architecture 完了後

**問い:**
- このアーキテクチャで開発を始めてよいか？
- 変更コストが高くなる前に、見直しが必要な箇所はないか？
- Binding Decisions の内容に納得しているか？

**判断基準:**
- `development-os/` のアーキテクチャ原則との整合
- S09 の品質基準を将来満たせるか

**選択肢:**
- **凍結（Freeze）** → S12 Roadmap へ。以降のアーキテクチャ変更は ADR 必須
- **修正して再確認** → 問題箇所を修正してから凍結

**記録先:** `governance/BINDING_DECISIONS.md`（アーキテクチャ凍結エントリ）

**タイミング:** S12 Roadmap + S13 APP STARTER OS 完了後

**問い:**
- PR-001 から着手してよいか？
- リソース（時間）の見積もりは現実的か？
- `05_FOUNDER_WEEKLY_SYSTEM.md` の時間配分を守れるか？

**判断基準:**
- MVP スコープが「小さすぎる・大きすぎる」ないか
- 週の開発時間で何ヶ月かかるかの見積もり

**記録先:** `governance/DECISION_LOG.md`

## CP-05 — β公開（MVP 完成後）

**タイミング:** S14 PR Development の MVP 完成後

**問い:**
- 「恥ずかしいが出せる」レベルに達しているか？（Reid Hoffman基準）
- 初期ユーザー（β）を招待する準備ができているか？
- 壊れても大丈夫な規模でリリースするか？

**判断基準:**
- `08_DEFINITION_OF_DONE.md` の全条件クリア
- `OPERATING_PRINCIPLES.md` #7 Long-term Quality（「恥ずかしいものは出さない」とのバランス）

**選択肢:**
- **β公開** → クローズドβとして限定公開
- **追加実装** → まだ必要な機能がある場合
- **正式公開** → βなしで直接正式公開

**記録先:** `governance/DECISION_LOG.md`

- 利用規約・プライバシーポリシーが整備されているか？

**判断基準:**

- βユーザーからの Retention が目標値に近いか

**記録先:** `governance/DECISION_LOG.md`

## CP-07 — 価格改定

**タイミング:** 正式公開後、数ヶ月の運営データ蓄積後

**問い:**
- 現在の価格がユーザーに受け入れられているか？
- LTV と CAC のバランスは取れているか？
- 値上げできるほどの価値証明ができているか？

**判断基準:**
- `business-os/` の価格設計原則
- `analytics-os/` の収益系 KPI

**選択肢:**
- 価格維持
- 値上げ（既存ユーザーへの移行措置を検討）
- プラン構成の見直し

**記録先:** `governance/DECISION_LOG.md` + `governance/BINDING_DECISIONS.md`

## CP-08 — PMF 判定

**タイミング:** 正式公開後 3〜6ヶ月

**問い:**
- 30日 Retention が目標値を達成しているか？
- 「このプロダクトがなくなったら困る」ユーザーが存在するか？
- NSM が自然に成長しているか？

**判断基準:**
- `analytics-os/` の PMF 基準（30日Retention 等）
- Sean Ellis 基準（40% が「非常に残念」と答えるか）

**選択肢:**
- **PMF 確認** → Growth フェーズへ。GTM を強化する
- **Pivot** → 機能・ターゲット・価格を変更する
- **Stop** → `business-os/` の撤退基準に照らして判断

**記録先:** `governance/DECISION_LOG.md`

## CP-09 — 海外展開

**タイミング:** 国内 PMF 確認後

**問い:**
- 国内市場が成熟しているか？
- 海外ユーザーからの自然な問い合わせが来ているか？
- 多言語対応のコストに見合うか？

**判断基準:**
- `research-os/` の市場分析
- `development-os/` の国際化対応コスト見積もり
- `OPERATING_PRINCIPLES.md` #8 Small Footprint（複雑さを増やさない）

**選択肢:**
- 海外展開する（英語化 + 海外GTM）
- 国内集中を継続
- 海外ユーザーを自然受け入れのみ（多言語対応なし）

**記録先:** `governance/DECISION_LOG.md`

**判断基準:**

## CP-11 — プロダクト終了（Exit Criteria 発動）

**タイミング:** `business-os/` の撤退基準に触れたとき

**問い:**
- MRR が X ヶ月連続で目標の Y% 以下か？
- 30日 Retention が Z% 以下が N ヶ月続いたか？
- Founder として誇りを持って続けられるか？

**選択肢:**
- **終了** → ユーザーへの告知・データエクスポート提供・サービス停止
- **売却** → 買い手を探す（タイムライン：最低 3ヶ月）
- **維持縮小** → 機能開発停止、メンテナンスのみで収益を維持

**記録先:** `governance/DECISION_LOG.md`（重要意思決定）

**判断基準:**
- `06_PORTFOLIO_MANAGEMENT.md` の同時開発数上限

**記録先:** `governance/DECISION_LOG.md`
