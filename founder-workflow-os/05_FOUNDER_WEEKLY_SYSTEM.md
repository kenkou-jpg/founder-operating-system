# 05 — Founder Weekly System

> 週45時間以内を前提とした、Founder の標準週間ルーティン設計。
> これはテンプレートです。自分のリズムに合わせてカスタマイズしてください。

---

## 設計原則

1. **週45時間が上限** — `FOUNDER_PHILOSOPHY.md` の Founder Health 原則に従う
2. **クリエイティブな仕事を最良の時間帯に** — 開発・設計は集中力が高い時間に
3. **ルーティンは固定、中身は変動** — 「何をするか」は固定、「何に集中するか」は変動
4. **週次レビューは省略しない** — 省略が一番コストが高い

---

## 標準週間ルーティン

### 月曜（週次スタート）— 9h

**AM: 週次レビュー（2h）**
- 先週の KPI 確認（`analytics-os/` のダッシュボード）
- 先週の完了タスク・未完了タスクの確認
- ブロッカー・懸念事項の洗い出し
- 今週の最重要タスク 3つを決める（`operations-os/` の週次レビュー）

**AM〜PM: 開発（5h）**
- 本週の PR 開発 / 設計作業
- 最も集中力を要するタスクをここに置く

**PM: 運営確認（2h）**
- 問い合わせ・フィードバックの確認・返信（`customer-success-os/`）
- エラー・アラートの確認

---

### 火曜 — 9h

**ALL: 開発集中日（8h）**
- PR Development（`development-os/pr-generator-os/` に従って実施）
- 割り込みを最小化する日

**PM: 数字確認（1h）**
- DAU / MAU の確認
- 前日からの変化を記録

---

### 水曜 — 9h

**AM: 開発（4h）**
- 火曜からの継続

**PM: ユーザー理解（3h）**
- ユーザーインタビュー（月2〜4回の枠）
- フィードバックの分析・整理（`customer-success-os/`）
- Research インサイトの記録（`research-os/`）

**PM: 改善設計（2h）**
- フィードバックから改善仮説を立てる
- 次の改善PRの Input Sheet を埋める

---

### 木曜 — 9h

**AM: 開発（4h）**
- 改善PR の実装 / または新機能PR

**PM: ブランド発信（3h）**
- note 記事の執筆 または 下書き
- X（Twitter）への投稿・交流
- `brand-os/` のブランドボイスに従って発信

**PM: テンプレート更新（2h）**
- 今週の開発で得たパターンを `templates/` や Founder OS に反映
- `knowledge-os/` へのナレッジ記録

---

### 金曜 — 9h

**AM: 開発（3h）**
- 今週の PR の仕上げ・テスト

**AM〜PM: 改善・自動化（3h）**
- 繰り返し作業の自動化機会の特定（`automation-os/`）
- 既存自動化の確認・改善

**PM: 週次クローズ（3h）**
- 今週の PR の Completion Report 提出
- `11_PR_RESPONSIBILITY_REGISTRY.md` の更新
- 来週の準備（タスクリスト・PR Input Sheet の準備）
- `governance/DECISION_LOG.md` への今週の決断の記録

---

## 日次 15 分ルーティン（毎朝）

毎朝 15 分、以下を確認する。

```
□ エラー・アラートの確認（`automation-os/` の監視）
□ 昨日の NSM 確認
□ 問い合わせ・レビューの確認（24時間以内返信ルール）
□ 今日の最重要タスク 1つを確認
```

## 月次 2h ルーティン（毎月第1月曜）

```
□ MRR / ARR の確認
□ 月次 Retention の確認
□ チャーン率の確認
□ 月次コスト確認（`finance-os/`）
□ `portfolio-os/PORTFOLIO_KPI.md` の更新
□ 今月の主な学びを `knowledge-os/` に記録
□ 来月の優先テーマを決める
```

## 四半期 半日ルーティン（3ヶ月ごと）

```
□ NSM・KPI の四半期振り返り
□ Business Strategy の見直し（`business-os/`）
□ Founder OS の見直し・更新（`governance/CHANGELOG.md`）
□ Founder Philosophy の確認（`FOUNDER_PHILOSOPHY.md`）
□ 次の四半期の重点テーマを決める
□ Decision Checkpoints の振り返り（`03_DECISION_CHECKPOINTS.md`）
□ Portfolio 状態の確認（各プロダクトのフェーズ確認）
```

## 週次システムの守り方

### 時間を守る仕組み

1. **朝に今日のタスクを 1つだけ決める** — 「これさえできれば今日は OK」のタスク
2. **開発時間はカレンダーでブロックする** — 予定として扱う
3. **ブランド発信は時間を決めてやる** — 「気が向いたら」では続かない
4. **金曜 PM は週次クローズ専用** — 新しいタスクを入れない

### 45時間を超えないための判断

- **週40h を超えたら:** 何かを削る。削るのは「新機能」か「発信」から
- **週45h を超えたら:** 必ず来週以降のどこかで休む。借金は返す
- **連続して45h 超えたら:** ペースを見直す。`FOUNDER_PHILOSOPHY.md` の Founder Health 参照

---

## 週次システムのカスタマイズ

このルーティンは **テンプレート** です。
実際の生活リズム・エネルギーパターンに合わせて調整してください。

変更してよいもの:
- 曜日・時間の配分
- 作業の順番

変更してはいけないもの:
- 週次レビュー（省略禁止）
- 日次 15 分の KPI 確認（省略禁止）
- 月次レビュー（省略禁止）
- 45時間の上限（原則）
